## Technical Implications for Building Juno for Maquinalista

### 1. DSA Attribution (biggest challenge)
With DSA, there are no manually chosen keywords. Google auto-targets based on website pages and generates search terms dynamically. Juno needs to:
- Pull **search terms** (not keywords) from the Google Ads API for DSA campaigns
- Attribute deals to search terms rather than bid keywords
- Handle the fact that `utm_term` may or may not be populated by DSA — needs research

### 2. PMAX Attribution
Performance Max gives very limited search term data. Google intentionally restricts visibility. Juno should:
- Pull whatever search term data PMAX exposes via API
- Flag PMAX spend that can't be attributed (transparency about the black box)
- This could be a differentiator — surfacing PMAX data that Google hides

### 3. RD Station CRM Integration (new — not HubSpot)
- Need to research RD Station API: can we pull contacts with UTM fields? Can we pull deals/opportunities and their associated contacts?
- RD Station is the #1 CRM in Brazil — building this connector opens the entire Brazilian market
- API docs: https://developers.rdstation.com/

### 4. Offline Sales Cycle
- The "deal" is not a website transaction — it's a phone/WhatsApp conversation that leads to a purchase recorded in RD Station
- Juno needs to read the deal record from RD Station, not track a website conversion
- First-touch attribution still works: keyword → UTM → contact record → deal in CRM

### 5. Victor's Server-Side Data
- They already have per-user navigation data with UTM tracking
- Potential opportunity: if RD Station integration is slow, Victor's server data could be an alternative data source
- Risk: Victor might think "we could build this ourselves" — address proactively by showing the Google Ads side of the join (keyword spend data + search term reports) which they don't have
